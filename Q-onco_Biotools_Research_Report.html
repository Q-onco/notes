Section 02

Molecular Visualization

Why Mol*, NGL, 3Dmol.js, and Speck are built the way they are — and what it tells us about our own renderer
Context

Protein structure visualization is essential for understanding BRCA1/2 mutations, PARP inhibitor binding modes, and HGSOC-relevant protein complexes. The question is not whether we need this — it is which architecture lets us build the most powerful, context-aware version.

🧊
Mol* (MolStar)
molstar.org · MIT · ~2MB gzipped · npm: molstar · RCSB PDB standard

Why it is built this way

Mol* was built by David Sehnal, Alexander Rose, and the RCSB PDB/PDBe teams to replace Jmol as the standard web viewer. Every design decision traces to one constraint: it must handle the entire PDB at any scale. The PDB's largest entries have 100M+ atoms (virus capsids, ribosomes). Most viewers collapse at this scale. Mol* does not, because of three architectural commitments:

1. Custom WebGL2 renderer (instanced geometry). Rather than wrapping Three.js, Mol* wrote their own GPU engine. General-purpose engines use a scene graph: 50,000 atoms = 50,000 node updates per frame = 50ms = below 20fps. Mol* uses instanced rendering — one sphere geometry on the GPU, rendered 50,000 times via a single draw call with per-instance data in a GPU buffer. This is why Mol* renders the ribosome at 60fps and Three.js-based viewers cannot.

2. Plugin/Behavior architecture (dependency injection). A central PluginContext holds a registry of behaviors. Every feature registers itself. We can write HGSOC-specific behaviors (BRCA1 mutation highlight, PARP binding pocket annotation) as plugins without forking the core library at all.

3. MOLJ state serialization. Mol* serializes complete viewer state (camera, selections, colorings, annotations) to JSON. For Q-onco: annotate a structure → save as MOLJ → AES-256 encrypt → GitHub .enc storage. Resume the exact same view on any device.

✓ Strengths

  • Handles any PDB size including viral capsids
  • Full plugin API — extend without forking
  • MOLJ state serialization — saveable sessions
  • All PDB representations: surface, cartoon, ball-and-stick
  • Electron density map support (2Fo-Fc, Fo-Fc)
  • Built-in sequence viewer synced to 3D
  • MIT licensed, actively maintained by RCSB

✗ Limitations for Q-onco

  • ~2MB gzipped — heavy first load
  • Steep learning curve on the plugin API
  • No 16-bit multi-channel TIFF (microscopy)
  • Generic UI — not tuned to cancer biology
  • No built-in HGSOC annotation database
→ Q-onco strategy

Phase 1: Embed Mol* with a stripped plugin set. Write a HGSOCAnnotationBehavior that auto-highlights known BRCA1/2 pathogenic variants when a structure loads. Serialize state to our encrypted GitHub storage. Bundle size: ~1.2MB with tree-shaking.

🔷
NGL Viewer
nglviewer.org · MIT · ~800KB gzipped · npm: ngl

NGL was built by Alexander Rose as his PhD project at UCSF. Uses Three.js under the hood — simpler API than Mol*, but inherits the scene graph overhead. Its real strength: 30+ file format parsers built-in, including trajectory formats for MD simulation playback. For HGSOC proteins (200–2000 residues), Three.js performance limitations are never a problem.

✓ Use for

  • Simple embeds where Mol* is overkill
  • MD trajectory playback
  • Projects needing a simpler API

✗ Skip because

  • Less actively maintained post-Mol*
  • No state serialization
  • No plugin system for HGSOC customization
⚛
3Dmol.js
3dmol.csb.pitt.edu · BSD-3 · ~600KB · npm: 3dmol · PDBQT native

Built at University of Pittsburgh for computational chemistry workflows. Its unique strength: native PDBQT support — AutoDock's output format. Every other viewer requires conversion from PDBQT to SDF/MOL2 before display. 3Dmol.js reads it directly, making it the essential companion to our Vina WASM docking pipeline.

→ Q-onco role

Use 3Dmol.js exclusively for the docking results panel. After Vina WASM runs (BRCA1 + olaparib/niraparib/rucaparib), render the PDBQT output poses natively — no format conversion step, no extra dependency.

✨
Speck
github.com/wwwtyro/speck · MIT · ~200KB · The AO sphere trick

Speck is built around one beautiful insight: ambient occlusion (AO) makes molecules look photorealistic, but real-time AO is expensive. Speck's solution: a software rasterizer running in a Web Worker pre-computes AO contribution for each atom's sphere, uploads the result as a WebGL texture. The GPU then texture-maps the pre-computed AO — separating compute (CPU, background) from display (GPU, main thread). Result: strikingly beautiful sphere renders that look like CGI paper figures, from a 200KB library.

Limitation: space-filling spheres only. No cartoon ribbons. Ideal for small molecules and ligands, not protein secondary structure.

→ Q-onco role: publication-quality drug molecule renders

Render olaparib, niraparib, rucaparib, veliparib with Speck's AO sphere mode. Parse SMILES/SDF with RDKit.js → pass XYZ coordinates to Speck → result looks like a polished publication figure ready for a grant submission.

Section 03

Microscopy & Image Analysis

Napari · QuPath · CellProfiler · Cellpose · StarDist · ilastik — architecture teardown and browser viability
HGSOC Microscopy Context

HGSOC microscopy data includes TMA cores, multiplexed IF with 6–10 channels, H&E whole-slide images at 40× (gigapixel), and live-cell imaging for drug response. All existing tools were designed for local computing. Our mission: bring this to the browser — and for cell segmentation and nucleus detection, we can do it today.

🔬
Napari
napari.org · BSD · Python/Qt desktop · Built by Chan Zuckerberg Initiative

Why it is built this way

Napari was built to replace FIJI/ImageJ for Python-centric researchers who needed to visualize arrays coming directly from NumPy/Dask pipelines. Its architecture: a visualization layer over in-memory arrays, rendered via VisPy (OpenGL under Qt). The layer model is Napari's core innovation:

Image + Labels + Points + Shapes + Vectors + Tracks — all composited in the same coordinate space. A researcher can load a raw IF image, overlay Cellpose masks, add manually annotated centroids, and draw ROI boundaries — all registered together. This layer architecture is exactly what we should adopt in our custom viewer.

✓ Architecture lessons

  • Layer model: Image + Labels + Points + Shapes → adopt this
  • Coordinate system first: all layers register to shared coords
  • Contrast limits per channel: essential for IF data
  • Plugin widgets: each tool adds its own UI panel

✗ Why not for us

  • Python/Qt desktop — zero browser portability
  • Requires local Python install + dependencies
  • No collaboration or sharing model
  • Can't run on web without a compute server
🧫
Cellpose
cellpose.readthedocs.io · BSD · U-Net gradient field model · ~25MB ONNX · WebGL EP ready

The gradient field trick

Built by Carsen Stringer and Marius Pachitariu at Janelia Research to solve the hardest problem in cell segmentation: overlapping, touching cells that classical watershed algorithms cannot separate. Traditional approaches fail when two cells touch because there is no intensity boundary between them.

Cellpose's solution: instead of predicting cell boundaries, it predicts a gradient field — a vector at every pixel pointing toward the cell's center of mass. Then Euler integration follows the gradient from every pixel to find cell centers. Cells that are spatially separate flow to different centers; touching cells flow to their respective centers. Segmentation falls out of the flow dynamics, not boundary detection — a fundamentally different and far more robust approach.

The network outputs two gradient channels (X and Y) plus a cell probability map. Exported to ONNX (~25MB), it runs with onnxruntime-web using the WebGL Execution Provider — GPU acceleration entirely in-browser — at approximately 2–5 seconds per 512×512 image on a mid-range laptop GPU.

// Browser Cellpose — full pipeline
const session = await ort.InferenceSession.create('/models/cellpose_cyto.onnx', {
  executionProviders: ['webgl'],  // GPU via WebGL2
});
const input = preprocessImage(imageData, 512, 512);
const feeds = { input: new ort.Tensor('float32', input, [1, 2, 512, 512]) };
const { output } = await session.run(feeds);
const masks = postprocess_flows(output);  // gradient integration → cell masks
→ Q-onco role: TMA cell segmentation

Upload TMA core image → Cellpose in-browser (WebGL EP) → cell masks generated → feature extraction (area, intensity per channel) → H-score computed → export to manuscript. Entire pipeline offline. No data leaves the browser.

⭐
StarDist
github.com/stardist/stardist · BSD · EPFL/EMBL · ~15MB ONNX · 2D_versatile_he model

Star-convex polygons for nuclei

StarDist was built at EPFL/EMBL as a specialized nucleus segmenter. The key insight: nuclei are roughly star-convex — any nucleus shape can be described by radial distances from the center to the boundary at N angles. StarDist predicts a 32-point star polygon for each nucleus rather than a pixel-wise mask, giving three advantages: better accuracy (star-convex constraint prevents "octopus" artifacts), polygon output directly usable as vector ROIs, and the 2D_versatile_he model specifically trained on H&E pathology images — the most common HGSOC histology stain.

→ Q-onco role: H&E nucleus detection

StarDist (2D_versatile_he, ~15MB ONNX) for H&E slide analysis. Researcher loads H&E image → nuclei detected with polygon boundaries → polygons stored as Shapes layer → overlay with IHC scoring → tumor purity assessment.

AI/ML Predictions

AlphaFold · ESM-2 · Vina WASM · DiffDock · PLIP — proteins, docking, and variant scoring in the browser
The browser ML revolution

Two years ago, saying "run AlphaFold in the browser" was absurd. Today the entire prediction stack is achievable: AlphaFold structures via free EBI API, ESM-2 150M parameter model running on WebGL EP, Vina WASM for full docking pipelines, and PLIP-equivalent geometry for interaction profiling. The compute exists. We need to orchestrate it.

🔭
AlphaFold Database (EBI API)
alphafold.ebi.ac.uk · Free · No auth · ~200M structures · pLDDT + PAE

AlphaFold2 has pre-computed structures for ~200 million proteins, served via a free, unauthenticated REST API. For any UniProt accession: predicted 3D coordinates (mmCIF/PDB), per-residue confidence (pLDDT 0–100), and predicted aligned error matrix (PAE — which domains are reliably positioned relative to each other).

For HGSOC research: BRCA1 (P38398), BRCA2 (P51587), PALB2 (Q86YC2), RAD51 (Q06609), PARP1 (P09874), PARP2 (Q9UGN5) — all pre-computed, immediately accessible, zero GPU, zero waiting. A researcher types a gene name, we query UniProt for the accession, query AlphaFold for the structure, load it into Mol*.

const resp = await fetch(
  `https://alphafold.ebi.ac.uk/api/prediction/${uniprotId}`
);
const [entry] = await resp.json();
// entry.cifUrl   → mmCIF structure → Mol*
// entry.paeImageUrl → PAE heatmap → sidebar
// entry.plddt    → per-residue confidence
→ Q-onco: HGSOC gene quick-load

Pre-populate: BRCA1, BRCA2, PALB2, RAD51C/D, BARD1, ATM, CHEK2, CDK12, PARP1, PARP2, CCNE1, TP53. One click → structure in Mol* → pLDDT confidence coloring → known pathogenic variants highlighted in red. Under 5 seconds, no server, no cost.

⚗️
AutoDock Vina (WASM)
vina.scripps.edu · Apache 2.0 · vina-wasm + openbabel-wasm + RDKit.js · Full in-browser docking

Why C++ and why WASM

AutoDock Vina was written in C++ because molecular docking requires extreme computational performance: scoring function evaluation for thousands of poses, Monte Carlo optimization, genetic algorithms. Equivalent JavaScript would take hours where C++ takes seconds. The WASM path preserves this performance: Vina compiled with Emscripten runs at ~70–80% of native C++ speed in the browser.

Three WASM libraries work together in sequence:

RDKit.js: Drug SMILES/SDF input → 3D conformer generation → geometry minimization → mol2 output

OpenBabel WASM: Format conversion mol2 → PDBQT (AutoDock-specific atom typing required by Vina)

Vina WASM: Receptor PDBQT + ligand PDBQT → docking → PDBQT poses with affinity scores (kcal/mol)

3Dmol.js: PDBQT poses rendered natively, binding pose comparison

→ Q-onco: PARP inhibitor binding analysis

Dock olaparib, niraparib, rucaparib, and veliparib against PARP1 catalytic domain. Get binding affinities and poses. Visualize in 3Dmol.js. Include in grant application. Entirely in-browser, entirely free, entirely private — no data upload.

🧬
ESM-2 (Evolutionary Scale Modeling)
Meta AI · MIT · 150M params → ONNX ~600MB · WebGL EP · 30ms/residue

Protein sequences as language

ESM-2 is a transformer language model trained on 250 million protein sequences. The key insight: protein sequences are a language, and the statistical patterns that make BERT powerful for text also capture evolutionary constraints in proteins. A missense mutation that is "grammatically wrong" in the protein language is likely deleterious — and ESM-2 quantifies exactly how wrong via log-likelihood ratio.

The 150M parameter model exports cleanly to ONNX (~600MB). With the WebGL Execution Provider in onnxruntime-web, inference runs at ~30ms/residue on a mid-range GPU. A 1800-residue BRCA1 protein takes ~54 seconds for full per-position scoring — acceptable for a background task with a progress indicator.

→ Q-onco: BRCA1/2 VUS pathogenicity scoring

Researcher pastes a variant: BRCA1 p.C61G. ESM-2 runs background (WebGL EP), returns log-likelihood ratio score compared to a pre-computed threshold distribution. Result: "Scores −4.8, consistent with pathogenic variants (>95th percentile harmful). Cysteine at position 61 participates in RING domain zinc coordination." Linked to the Mol* structure view.

Genomics Browser

IGV.js teardown — HTTP range requests, BAM/VCF handling, and HGSOC somatic mutation visualization
HGSOC genomics context

High-grade serous ovarian cancer has one of the most complex genomic landscapes in oncology: near-universal TP53 mutation, BRCA1/2 in ~20%, RAD51C/D/BARD1 in ~10%, CDK12 in ~3%, CCNE1 amplification in ~20%, and pervasive copy number instability. Visualizing these variants in their genomic context is essential for understanding resistance mechanisms and preparing grant narratives.

🔬
IGV.js
igv.org · MIT · ~2MB · npm: igv · Used in GDC, cBioPortal, TCGA

The HTTP range request trick

The central challenge: human genome reference is 3GB. A WGS BAM file is 100GB. No browser can load these files in full. IGV.js solves this with HTTP range requests (RFC 7233) — it fetches only the bytes needed for the current view window.

BAM files use BGZF compression (blocked gzip) specifically to enable this: the file is divided into 65KB blocks, each independently compressed, with a BGZF index (.bai) mapping genomic coordinates to byte offsets. IGV.js fetches the .bai index once (small), uses it to calculate which blocks contain reads overlapping the current view, and fetches only those bytes via the HTTP Range header.

For local files, the File System Access API exposes the same interface: FileSystemFileHandle.getFile().slice(start, end). Drag-drop a BAM file from the local disk — IGV.js reads only the needed blocks, no upload, gigapixel data never leaves the machine.

// IGV.js with HGSOC pre-loaded tracks
const browser = await igv.createBrowser(container, {
  genome: 'hg38',
  locus: 'BRCA1:41,196,311-41,277,500',
  tracks: [
    { type: 'annotation', url: '/tracks/hgsoc_variants.vcf',
      name: 'HGSOC Hotspots', colorBy: 'FILTER' },
    { type: 'annotation', url: '/tracks/brca_exchange.bed',
      name: 'ClinVar Pathogenic', color: '#e94560' },
    { type: 'variant', url: 'patient.vcf', indexURL: 'patient.vcf.tbi',
      name: 'Patient Variants', filter: { minGQ: 30 } },
  ]
});

  • Production-grade, used in GDC, cBioPortal
  • HTTP range requests — handles 100GB BAMs
  • Local file drag-drop, no upload needed
  • VCF, BAM, BED, bigWig, bigBed, seg, MAF
  • hg38/hg19 reference genomes built-in
  • Variant coloring by consequence/filter

✗ Limitations

  • ~2MB bundle — heavier tab initial load
  • No native scRNA-seq marker visualization
  • No direct link to protein structure viewer
  • CORS headers required for remote BAMs
  • No built-in HGSOC annotation tracks

Section 06

Why Tools Are Built the Way They Are

The five architectural patterns underlying every design decision across all 18 tools

After analyzing 18 tools, every design decision traces back to one of five fundamental constraints. Understanding these explains why tools that seem overcomplicated are exactly as complex as they need to be — and where Q-onco can be smarter.

🎨
Pattern 1: The GPU Bottleneck → Custom Renderers

General-purpose 3D engines (Three.js, Babylon.js) use a scene graph: a tree of objects each with a transform, material, and geometry. Updating 50,000 atoms = 50,000 node updates per frame. Even at 1μs per node: 50ms per frame = below 20fps. This is why a naive Three.js protein viewer grinds to a halt at 100,000 atoms.

High-performance scientific viewers use instanced rendering: one geometry definition uploaded to the GPU once, rendered N times with per-instance data (position, color, radius) in a GPU buffer. Draw calls: 50,000 → 1. Mol* renders 1M atoms at 60fps using this technique.

Q-onco implication

If we write a custom molecular renderer, we must use instanced rendering from day one. Three.js is the wrong base. Write directly against WebGL2 — or use a thin abstraction like regl or twgl.js that exposes instancing without a scene graph overhead.

⚡
Pattern 2: The Main Thread Rule → Web Workers Everywhere

The browser's main thread handles rendering, UI events, and JavaScript execution. Any computation over ~16ms causes a dropped frame; over ~100ms causes a UI freeze users perceive as a hang. Parsing a 10MB PDB file: ~500ms. Running Cellpose inference: 5 seconds. Parsing a 50MB BAM block: ~200ms. All of these block the UI if run on the main thread.

All performance-critical tools use Web Workers: background threads with their own JS context. Architecture: Worker parses/computes → postMessage result to main thread → WebGL renders. The key nuance: SharedArrayBuffer + Atomics allow zero-copy transfer — critical for passing large image arrays to ONNX runtime. Requires COOP/COEP HTTP headers in our Vite config.

🧩
Pattern 3: Plugin Systems → Five-Year Survivability

Mol*, Napari, and IGV.js all use plugin/extension architectures. This is not over-engineering — it is the only way a research tool survives 5+ years: researchers always need customization that the core team cannot anticipate. Mol*'s dependency injection model: a PluginContext holds a registry of behaviors, each observing state changes and reacting.

// Writing a Mol* HGSOC annotation behavior
const HGSOCAnnotations = PluginBehavior.create({
  name: 'hgsoc-annotations',
  ctor: class extends PluginBehavior.Handler {
    register(): void {
      this.subscribeObservable(
        this.plugin.state.data.events.changed,
        async () => {
          const structure = this.plugin.managers.structure
            .hierarchy.current.structures[0];
          const id = await fetchUniprotId(structure);
          if (HGSOC_PROTEINS.includes(id))
            await highlightPathogenicVariants(this.plugin, id);
        }
      );
    }
  }
});
📦
Pattern 4: File Format Complexity → Reuse Parsers

Scientific file formats encode decades of domain knowledge. PDB format (1971): 132-column fixed-width records. mmCIF (1990): self-describing key-value format. PDBQT: adds AutoDock-specific atom typing. BGZF: blocked gzip with an external index. Each format has edge cases, optional fields, and implementation bugs in producing software.

Practical rule: format parsers are infrastructure, not features. Reuse existing ones. Mol*'s mmCIF parser, cheminfo's SDF parser, bgzf-browser for BAM. The one exception: 16-bit TIFF with custom channel metadata (Fiji/OMERO format) — no browser library handles this correctly. We write our own decoder.

💾
Pattern 5: State Serialization → The Reproducibility Problem

A researcher annotates a protein structure for a paper: selected residues, custom coloring, camera angle, labels. They close the browser. Tomorrow, they need the exact same view for a revised figure. Without state serialization, this work is lost. This is why Mol*'s MOLJ format exists — and why we must implement it.

For Q-onco: viewer state (MOLJ) + analysis results (cell masks, variant scores) + researcher annotations → single session JSON → AES-256 encrypted → .enc file in GitHub. A session started on a laptop can be resumed on a different machine or shared with a collaborator who has the decryption key.

HGSOC Researcher Pain Points

What oncology researchers actually struggle with — mapped to our technical capabilities
🧬 Pain Point 1: BRCA1/2 Variant Interpretation
The Problem Today

A patient has BRCA1 c.1016dupA (p.T339Nfs*11) — a frameshift variant. Is this pathogenic? Which domain does it disrupt? Currently: search ClinVar, look at LOVD, open Chimera locally, find the residue manually in the sequence viewer, check the literature. 45+ minutes per variant.

Q-onco Solution

Type BRCA1 → AlphaFold loads in Mol* → HGSOC behavior auto-highlights frameshift position → domain annotation shown → ESM-2 score in sidebar → ClinVar classification pre-fetched → one-click export generates labeled structure figure. Under 2 minutes.

💊 Pain Point 2: PARP Inhibitor Resistance Mechanisms
The Problem Today

Patient progresses on olaparib. Reversion mutation in BRCA2? Secondary mutation restoring HR? PARP1 promoter hypomethylation? Each hypothesis requires different analyses across different tools that do not communicate with each other.

Q-onco Solution

Dock olaparib vs WT PARP1 + reversion mutant side by side → binding affinity comparison → ESM-2 scores for reversion mutations → genomics tab shows the reversion mutation in context → Enzo synthesizes resistance literature → grant narrative drafts itself.

🔬 Pain Point 3: TME Quantification from Multiplexed IF
The Problem Today

Measuring CD8+ TIL density, PD-L1 expression on tumor vs stroma, macrophage polarization markers. Currently: Fiji → QuPath → Excel → Prism. Four tools, three exports, manual data transfer at each step.

Q-onco Solution

Load multiplexed TIFF → toggle channels (DAPI/CD8/PD-L1/CK/CD68) → Cellpose segments tumor vs immune by channel profile → H-score per compartment auto-computed → results to Research tab → Enzo drafts the TME paragraph. Zero manual data transfer.

📊 Pain Point 4: TMA Scoring for Biomarker Studies
The Problem Today

Scoring a 200-core TMA for RAD51 foci (HR function marker): scan slides, download gigapixel images, set up QuPath project, define core boundaries, batch analysis, export scores, clean data. Typical: 2–3 weeks.

Q-onco Solution

Upload TMA TIFF → automatic core detection → Cellpose nuclei per core → channel intensity computed → H-score + Allred score automatic → CSV export linked to clinical metadata. Typical: 2–3 hours.

Section 09

Build vs Embed: Decision Framework

Tool-by-tool decisions — when to embed existing libraries, when to build custom
The decision rule

If an existing library with a free license does 90% of what we need, we embed it and extend it. We only build custom when the domain specificity makes existing tools fundamentally inadequate — or when the feature is unique enough to HGSOC research that no general tool will ever address it.

Protein structure visualization
EMBED Mol*
Does 100% of what we need. Plugin API covers all HGSOC customization. MIT license. Bundle tree-shakeable to ~1.2MB.
Small molecule rendering
EMBED Speck
200KB, ambient occlusion rendering, perfect for publication-quality drug molecule figures. No alternative needed.
Docking visualization
EMBED 3Dmol.js
Native PDBQT — essential for Vina output. Only browser tool that does this. BSD-3 license.
Cell segmentation
EMBED Cellpose
Pre-trained ONNX models, WebGL EP acceleration. No need to train our own. Gradient field approach is state-of-art.
Nucleus detection (H&E)
EMBED StarDist
2D_versatile_he model is state-of-art for H&E pathology. Star-convex polygons give vector ROIs directly.
Molecular docking
EMBED Vina WASM
Full C++ AutoDock Vina compiled to WASM at 70–80% native speed. No browser-native alternative exists.
Genome browser
EMBED IGV.js
Production-grade, used in GDC and cBioPortal. HTTP range requests for 100GB BAMs. Local file drag-drop. MIT.
Multi-channel IF viewer
BUILD CUSTOM
No existing browser tool handles 16-bit multi-channel TIFF. This is the critical domain gap. Custom WebGL2 with Napari's layer model architecture.
HGSOC annotation system
BUILD CUSTOM
Unique to our context — BRCA1/2 domain boundaries, pathogenic variant database, PARPi binding site overlays. No general tool will ever include this.
Variant pathogenicity scoring
EMBED ESM-2 ONNX
150M param protein language model, WebGL EP. State-of-art for zero-shot variant effect prediction. Free, MIT.
AlphaFold structures
API ONLY
EBI serves 200M pre-computed structures free, no auth. Zero compute cost. Never run locally when EBI has done it for us.
Protein-ligand interactions
BUILD PLIP-lite
PLIP (the Python reference) is 500 lines of geometry math. ~500 lines of equivalent JS geometry covers 90% of use cases without a Python server.
